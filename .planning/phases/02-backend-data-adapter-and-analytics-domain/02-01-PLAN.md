# Plan 02-01: Harden backend contract validation

<objective>
Make the backend analytics layer fail predictably on bad upstream responses and
expand test coverage from pure analytics helpers into orchestration and route-level
contracts.
</objective>

<tasks>
## Task 1
<read_first>
- backend/app/services/decision_service.py
- backend/app/clients/alldevice.py
- backend/app/main.py
</read_first>
<action>
Add explicit validation helpers in `DecisionService` so downtime endpoints require
`success: true` and a list-shaped `response`, while task reports require a dict-shaped
`response` object. Raise stable backend errors instead of silently normalizing bad payloads.
</action>
<acceptance_criteria>
- `backend/app/services/decision_service.py` contains validation helpers for downtime and task reports
- failed or malformed downtime responses raise `UpstreamAPIError`
</acceptance_criteria>

## Task 2
<read_first>
- tests/backend/test_line_mapping_and_metrics.py
- backend/app/api/deps.py
- backend/app/main.py
</read_first>
<action>
Add backend tests that cover `DecisionService` orchestration and FastAPI route behavior
using stubs and dependency overrides. Include at least one successful route test and one
contract-failure test.
</action>
<acceptance_criteria>
- `tests/backend/test_decision_service_and_routes.py` exists
- `.venv\\Scripts\\python.exe -m pytest tests\\backend -q` exits 0
</acceptance_criteria>
</tasks>

<must_haves>
- Backend does not silently accept malformed upstream payloads
- Routes remain thin and continue using dependency injection
- Verification covers more than helper math
</must_haves>
